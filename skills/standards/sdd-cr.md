# SDD CR (Code Review with Spec Context)

## Purpose

Reviews code changes with full SDD context: reads the relevant OpenSpec spec files from `openspec/specs/modules/<module>.md`, compares the implementation against the spec's documented classes, APIs, and dependencies, and reports gaps. Unlike sdd-review (which checks coding style and security), sdd-cr focuses on **spec compliance** -- does the code implement what the spec says it should? Are there behaviors in the code not documented in the spec? Are documented APIs missing from the implementation?

## When to Use

- After implementing a change that has an associated spec file
- When reviewing a PR that touches modules with OpenSpec specs
- As part of the sdd-verify workflow to check spec-code consistency
- Before archiving a change to ensure the spec was properly updated
- When a spec was manually edited and you need to verify the code matches

## Instructions

1. **Identify affected specs** -- from the change proposal or from git diff, determine which modules were changed. Look up their spec files:
   - Module specs: `openspec/specs/modules/<module>.md`
   - Schema specs: `openspec/schemas/<table>.md`
   - Also check `CLAUDE.md` for the module map to cross-reference

2. **Read each spec** -- extract the documented contract:
   - **Classes** section: what classes/interfaces should exist, their descriptions
   - **API** section: what functions/methods should be exported, their signatures, their descriptions
   - **Dependencies** section: what other modules this module depends on
   - **Overview**: the module's documented purpose and responsibilities

3. **Review code changes** against the spec:
   - **Added code**: does it implement a documented requirement? If not, should the spec be updated?
   - **Modified code**: does the change preserve the documented API contract? If signatures changed, was the spec updated?
   - **Removed code**: was the documented API deprecated first? Is there a migration path?
   - **New exports**: are they documented in the spec's API section?

4. **Verify spec compliance** checklist:
   - [ ] All documented classes exist in the code
   - [ ] All documented APIs exist with matching signatures
   - [ ] All documented dependencies match actual imports
   - [ ] No undocumented public APIs added without spec updates
   - [ ] No spec-documented features removed without deprecation
   - [ ] Module responsibilities match the spec's Overview description

5. **Report gaps** with spec references:
   ```
   SPEC COMPLIANCE REPORT
   ======================

   Module: auth/login
   Spec: openspec/specs/modules/login.md

   GAPS (spec says, code doesn't do):
     - Spec documents `validateSession(token)` but function not found in code
     - Spec documents class `LoginController` but file contains `AuthController`

   DRIFT (code does, spec doesn't say):
     - Code exports `refreshToken()` but spec has no mention of token refresh
     - Code imports `db/cache` but spec's Dependencies section lists only `db/queries`

   MATCH (spec and code agree):
     - `authenticateUser(credentials)` — matches spec signature and description
     - `logoutUser(sessionId)` — matches spec signature and description

   SEVERITY: HIGH — 2 gaps, 2 drifts
   ```

6. **Recommend actions** for each gap/drift:
   - **GAP**: implement the missing feature OR update the spec to reflect current state
   - **DRIFT**: add the missing behavior to the spec OR remove the undocumented code
   - **MATCH**: no action needed

## Commands Used

- `sdd status` -- view spec counts and active changes
- `sdd sync` -- detect spec-code drift via sync state comparison
- `sdd doctor` -- checks sync state consistency (stale entries are a red flag for spec drift)

## Spec Structure Reference

Each module spec in `openspec/specs/modules/<module>.md` follows the template generated by `writeSpec()` in `cli/src/integrations/openspec.ts`:

```markdown
# Module: <title>

## Overview
<description>

## Classes
### <ClassName> — <description>

## API
### <signature> — <description>

## Dependencies
- <dep>
```

## Auto-Generated vs. Manual Specs

- **Auto-generated specs** contain the banner: `> Auto-generated by sdd-exoskeleton`
- **Manual specs** do not have this banner and will NEVER be overwritten by `sdd sync`
- When reviewing auto-generated specs, flag any behaviors that the auto-generator missed (stubs, placeholder content)
- When reviewing manual specs, verify the content is still accurate against the current code

## Integration with Other Skills

- **sdd-propose**: the proposal should reference the specs this change affects
- **sdd-apply**: implementer must update specs to match the implementation
- **sdd-verify**: runs sdd-cr as part of the verification step
- **sdd-archive**: ensures specs are finalized before archiving
- **sdd-review**: checks code quality; sdd-cr checks spec compliance

## Code Location

- Spec reading/writing: `cli/src/integrations/openspec.ts` (`writeSpec()`, `writeSchema()`)
- Spec generation: `cli/src/generators/specs.ts` (`generateModuleSpecs()`, `buildModuleSpec()`)
- Sync conflict detection: `cli/src/sync/delta.ts` (`detectConflicts()`)
