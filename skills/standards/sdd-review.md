# SDD Review

## Purpose

Performs an automated code review of changed files against the project's coding standards as documented in CLAUDE.md and enforced by the SDD Exoskeleton. The review checks: coding style (via language-appropriate linter), naming conventions, immutability patterns, security vulnerabilities, function and file size limits, error handling, and input validation. Results are categorized by severity: CRITICAL (block merge), HIGH (should fix), MEDIUM (consider), LOW (optional).

## When to Use

- After writing or modifying code (mandatory before commit)
- As part of the sdd-apply workflow before marking a change complete
- Before creating a pull request
- When security-sensitive code is changed (auth, payments, user data)
- As a pre-commit hook in CI/CD

## Instructions

1. **Determine changed files** -- use `git diff --name-only` against the target branch or `sdd sync` to get the list of files changed since the last sync point.

2. **Check coding standards** (for each changed file):
   - **Immutability**: scan for mutation patterns (`obj.field =`, `arr.push(`, `arr.sort(` without spread). Flag any direct mutation on parameters or shared state.
   - **Naming conventions**: verify camelCase for variables/functions, PascalCase for interfaces/types/components, UPPER_SNAKE_CASE for constants, `is`/`has`/`should`/`can` prefixes for booleans.
   - **Function size**: flag functions >50 lines. Recommend extraction.
   - **File size**: flag files >800 lines. Recommend splitting.
   - **Deep nesting**: flag code with >4 levels of nesting (if/for/while/try). Recommend early returns.

3. **Check security** (CRITICAL priority):
   - **Hardcoded secrets**: scan for API keys, passwords, tokens, connection strings. Block any finding.
   - **SQL injection**: flag string concatenation or interpolation in database queries. Require parameterized queries.
   - **XSS**: flag unescaped user input in HTML/template rendering.
   - **Path traversal**: flag unsanitized file paths using user input.
   - **Input validation**: flag functions that process user/external input without validation.

4. **Check error handling**:
   - Verify every `try` has a corresponding `catch` that handles the error (no empty catch blocks)
   - Verify async functions have error handling (`.catch()` or try/catch)
   - Verify error messages do not leak sensitive information (stack traces, internal paths)

5. **Check immutability** (from coding-style.md CRITICAL rule):
   - Flag `obj.field = value` on function parameters
   - Flag array mutation methods: `push`, `pop`, `splice`, `sort`, `reverse`, `shift`, `unshift`
   - Require spread operator or `structuredClone` for creating new objects
   - For TypeScript: verify `Readonly<T>` is used on function parameters that should not be mutated

6. **Check KISS/YAGNI/DRY**:
   - Flag premature optimization (complex code without profiling evidence)
   - Flag speculative features (code that is never called)
   - Flag copy-pasted blocks (>5 identical lines across files)
   - Suggest extraction when the same logic appears in 3+ places

7. **Report findings** with severity and file locations:
   ```
   CRITICAL (block):
     src/auth/login.ts:42 — Hardcoded API key on line 42

   HIGH (should fix):
     src/utils/helpers.ts:120-195 — formatUserData is 76 lines, exceeds 50-line limit

   MEDIUM (consider):
     src/db/queries.ts:88-95 — Deep nesting (5 levels) in buildFilterQuery

   LOW (optional):
     src/api/routes.ts:15 — Consider renaming 'data' to 'userData' for clarity
   ```

## Commands Used

- `sdd lint` -- run language-appropriate linter
- `sdd status` -- view current project state
- `git diff --name-only <base>...HEAD` -- list changed files

## Review Severity Levels

| Level | Meaning | Action |
|-------|---------|--------|
| CRITICAL | Security vulnerability or data loss risk | BLOCK - Must fix before merge |
| HIGH | Bug or significant quality issue | WARN - Should fix before merge |
| MEDIUM | Maintainability concern | INFO - Consider fixing |
| LOW | Style or minor suggestion | NOTE - Optional |

## Integration with Other Skills

- **sdd-cr**: focuses on spec compliance; sdd-review focuses on code quality
- **sdd-lint**: runs language-specific linters; sdd-review adds semantic checks
- **sdd-apply**: runs sdd-review after implementation before marking complete
- **sdd-verify**: includes sdd-review as part of the verification step

## Code Location

- Coding standards reference: `CLAUDE.md` (generated by `cli/src/generators/constitution.ts`)
