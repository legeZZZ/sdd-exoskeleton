# SDD Arch Guard

## Purpose

Loads the project's module topology and boundary map (generated by `sdd init` Phase 3/4), then validates proposed changes against architectural boundaries. Detects forbidden cross-boundary dependencies, new circular dependencies, and violations of the layering conventions encoded in the boundary structure. Acts as a pre-commit/pre-merge gate to prevent architectural drift.

## When to Use

- Before merging a PR that adds or changes module dependencies
- When proposing a change that touches multiple modules (run before sdd-apply)
- After a refactoring that moved code between modules (run before sdd-archive)
- When `sdd analyze` reports new circular dependencies
- As a CI check in your pipeline (run `sdd doctor` which includes structural validation)

## Instructions

1. **Load the module topology** -- read the `.sdd-exoskeleton/config.json` to find the project structure, then load the boundary map:
   - Boundaries are extracted by `identifyBoundaries()` in `cli/src/analyzers/structure.ts`
   - Each boundary groups modules under the same top-level directory (e.g., `src/auth/` -> boundary "auth")
   - Files at the source root belong to boundary "root"
   - `identifyCoreApis()` finds exports that cross boundaries

2. **Check proposed changes against boundaries** -- for each proposed change:
   - **New dependency**: verify the source and target are in the same boundary, OR that the cross-boundary dependency is documented and intentional
   - **New module**: verify it is placed in the correct boundary based on its directory path
   - **Removed module**: verify no orphaned references remain in other boundaries
   - **Moved module**: verify the move does not create a cycle or violate layer ordering

3. **Check for circular dependencies** -- the `detectCycles()` function in `cli/src/analyzers/structure.ts` uses three-colour DFS (white/gray/black) to find cycles in the dependency graph. Any new cycle is a BLOCK violation.

4. **Check for god modules** -- modules with >50 exports (`GOD_MODULE_EXPORT_THRESHOLD`) are flagged. A proposed change that adds exports to a module already near the threshold should be accompanied by a plan to split it.

5. **Report violations** with severity:
   - **BLOCK**: new circular dependency, cross-boundary dependency to a forbidden boundary, deleting a public API without deprecation
   - **WARN**: god module approaching threshold, deep nesting added (>4 levels), orphan module created
   - **INFO**: new module without a spec file, missing boundary documentation

6. **Resolve violations** before merging:
   - BLOCK issues must be fixed by restructuring the change
   - WARN issues should have a follow-up task filed
   - INFO issues are advisory

## Commands Used

- `sdd status` -- view current topology, dependencies, and boundaries
- `sdd doctor` -- runs structural health checks including cycle detection
- `sdd init . --dry-run` -- preview what the analyzer would detect without applying changes

## Boundary Map Example

```
Project: my-app
Boundaries (from src/):
  auth      -> modules: [login, register, session]
  db        -> modules: [connection, migrations, queries]
  api       -> modules: [routes, middleware, handlers]
  utils     -> modules: [format, validate, http]
  root      -> modules: [index, app, config]

Cross-boundary APIs:
  db.connection  -> import by api.routes, api.handlers
  utils.validate -> import by auth.login, auth.register
```

## Validation Rules

| Rule | Check | Severity |
|------|-------|----------|
| No cycles | DFS on dep graph | BLOCK |
| Boundary isolation | `extractBoundaryName()` comparison | WARN if unexpected |
| God module limit | `exports.length > 50` | WARN |
| Orphan prevention | In-degree + out-degree > 0 | INFO |
| Depth limit | `fileDepthAfterSrcDir() <= 4` | WARN |
| Public API stability | exports removed without deprecation | BLOCK |

## Code Location

- Boundary detection: `cli/src/analyzers/structure.ts` (`identifyBoundaries()`, `identifyCoreApis()`)
- Cycle detection: `cli/src/analyzers/structure.ts` (`detectCycles()` — DFS with three-colour marking)
- God module detection: `cli/src/analyzers/structure.ts` (`findGodModules()`)
- Orphan detection: `cli/src/analyzers/structure.ts` (`findOrphanModules()`)
- Depth check: `cli/src/analyzers/structure.ts` (`findDeepNestedFiles()`)
- Module topology type: `cli/src/analyzers/structure.ts` (`ModuleTopology`, `Boundary`, `HealthReport`)
