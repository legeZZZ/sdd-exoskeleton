# SDD Sync

## Purpose

Detects code changes since the last sync point, analyzes their impact on specs and the Obsidian vault, generates a delta change proposal under `openspec/changes/<timestamp>-sync/`, and updates `.sdd-exoskeleton/sync-state.json` with new file hashes. When used with `--watch`, runs continuously via chokidar file watching, debouncing changes before triggering sync.

## When to Use

- After any code change that should be reflected in OpenSpec specs or Obsidian notes
- Before a commit to ensure specs are up to date with implementation
- When `sdd doctor` reports stale sync entries
- When you want continuous sync during active development (`--watch`)
- As part of the SDD evolution cycle: propose (sdd-propose) -> apply (sdd-apply) -> verify (sdd-verify) -> sync (sdd-sync)

## Instructions

1. **Ensure a baseline exists** -- `sdd init` must have been run at least once, producing `.sdd-exoskeleton/sync-state.json` and the OpenSpec directory tree.

2. **Run `sdd sync`** -- the sync pipeline executes:
   a. **Detect changes**: compares the git diff between the last sync ref and HEAD. Uses `getChangedFiles()` from `cli/src/utils/git.ts`, then determines each file's status (added/modified/removed) by checking existence and content hash in both states.
   b. **Resolve symbols**: maps changed files to `SymbolChange` objects with `name`, `type`, `module`, `summary`. Currently infers symbols from file paths; future integration will use CodeGraph MCP for real symbol resolution.
   c. **Analyze impact**: computes impact level (low/medium/high) per symbol, maps each symbol to affected specs by scanning `openspec/specs/modules/` and `sync-state.json` entries.
   d. **Generate delta**: creates `openspec/changes/<timestamp>-sync/proposal.md` summarizing affected modules, specs, and changes.
   e. **Update sync state**: hashes each affected file and updates its `SyncEntry` in `.sdd-exoskeleton/sync-state.json` with the new hash and timestamp.
   f. **Detect conflicts**: checks whether any affected spec contains manual edits (no auto-generation marker). Reports conflicts that need manual resolution.

3. **Review the generated delta** -- open `openspec/changes/<timestamp>-sync/proposal.md` to see what changed, the impact level, affected modules, and affected specs.

4. **For continuous sync** -- run `sdd sync --watch` to start a chokidar-based file watcher on `src/**/*.{ts,tsx,js,jsx,py,go,...}`. Changes are debounced at 500ms (configurable) so rapid edits fire once with the full file list. Ignores `node_modules/`, `.git/`, `dist/`, `build/`, `.sdd-exoskeleton/`, `sdd-vault/`, `openspec/`.

## Commands Used

- `sdd sync` -- one-shot sync from last sync ref to HEAD
- `sdd sync --watch` -- continuous file watching with debounced sync
- `sdd sync --debounce <ms>` -- custom debounce window (default: 500ms)

## Output

- `openspec/changes/<timestamp>-sync/proposal.md` -- auto-generated change proposal with summary, impact level, affected modules/specs, and change list
- `.sdd-exoskeleton/sync-state.json` -- updated `SyncState` with new hashes and timestamps per tracked file

## Conflict Handling

When `detectConflicts()` finds a spec that was manually edited (does not contain "Auto-generated by sdd-exoskeleton"), the sync:
- Reports the conflict with file path and reason
- Does NOT overwrite the manually-edited spec
- Lists conflicts in the delta proposal under a "Conflicts" section

Resolve conflicts by:
1. Reviewing the delta proposal to understand what changed
2. Manually merging the auto-generated content into the manually-edited spec
3. Re-running `sdd sync` (the new hash will match since no further code changes)

## Sync State Structure

Each entry in `.sdd-exoskeleton/sync-state.json` tracks:
```json
{
  "entries": {
    "<module>": {
      "lastHash": "<sha256>",
      "lastSyncedAt": "<ISO 8601>",
      "mappedSpecs": ["<spec>.md"],
      "mappedObsidianNodes": ["<note>.md"]
    }
  }
}
```

The `diffState()` function in `cli/src/sync-state.ts` computes added, modified, and removed entries between two states.
